## 1. 깃허브 프로젝트 및 칸반 보드 현황 공유
* **칸반 보드 활용:**
    * 모든 이슈와 작업 내역(백로그)은 깃허브 프로젝트의 칸반 보드에 등록됨.
    * 작업 시작 시 `In Progress`로 이동시키고, 완료 시 `Done`으로 이동.
    * PR(Pull Request) 후 코드 리뷰가 필요한 경우 `In Review` 탭 활용.
* **WBS 및 일정 공유:**
    * `README.md` 하단에 개발 일정 및 WBS(작업 분할 구조도) 표 추가 완료.
    * 1주 차~2주 차에 해당하는 태스크들이 이슈 보드에 등록되어 있으며, 우측의 담당자(Assignee) 아이콘을 확인하여 본인에게 할당된 작업을 진행하면 됨.

## 2. 깃허브 협업 브랜치 및 PR(Pull Request) 규칙
* **이슈 및 PR 연동 (성욱님 질문 사항):**
    * 로컬로 다운로드 받은 폴더 구조(백엔드, 닥스 등)를 바탕으로 각자 작업 진행.
    * 자세한 작성 및 기여 규칙은 `CONTRIBUTING.md` 문서에 정리되어 있음.
* **브랜치 명명 규칙(Branch Naming Convention):**
    * `[작업분류]/[이슈번호]` 형식 사용.
    * 예시: 기능 추가 시 `feature/#16`, 버그 수정 시 `bug/#17`, 문서 작업 시 `docs/#22`.
* **PR 생성 프로세스:**
    1. 할당된 이슈 확인.
    2. 로컬에서 새로운 브랜치 생성 (`feat/#이슈번호`).
    3. 작업 완료 후 커밋 및 리모트 레포지토리로 푸시(Push).
    4. 깃허브에서 `dev` 브랜치를 타겟으로 PR 생성.
    5. 리뷰어의 확인 후 `dev` 브랜치에 병합(Merge).

## 3. 프론트엔드 UI/UX 설계 및 진행 상황 리뷰 (시연님)
* **현재 진행 상황:** UI/UX 디자인 및 퍼블리싱 작업 진행 중 (HTML/CSS 뼈대 구축 완료, API 및 소켓 연동 작업 대기 중).
* **와이어프레임(Wireframe) 리뷰:**
    * 총 3개의 페이지 구성: Aider 에이전트 뷰, 메인 대시보드, 더미 AI 에이전트 뷰(Claude, Gemini 등 실시간 상태 확인 창).
* **피드백 사항 (진행자):**
    * **더미 에이전트 페이지 수정:** 화면을 4등분 하기보다는, 시각화(데이터/그래프 등) 영역을 더 크게 확보하고 하단의 HTML 코드가 들어갈 부분을 조정하여 화면 배치(Layout)를 개선할 것을 권장.
* **프론트엔드 작업 PR 방향:** * 현재 작업 중인 브랜치에서 마무리 후 `dev` 브랜치로 PR 요청. 
    * 뼈대 및 HTML/CSS 로직만 먼저 PR로 올리고 충돌이 없다면 병합 진행 예정.

* **원인 분석:**
