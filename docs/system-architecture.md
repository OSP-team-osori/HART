# 🏛️ System Architecture & Design Decisions

본 문서는 HART 프로젝트의 핵심 기술 아키텍처와 기술적 의사결정(ADR)의 배경을 명세합니다.

## 1. 실시간 로그 파이프라인 (Real-time Log Streaming)

### 1.1. 파일 I/O 배제 및 Stdout 가로채기
- **의사결정:** AI 에이전트의 구동 내역을 가져올 때 `.aider.chat.history.md` 등의 파일 읽기(File I/O) 방식을 사용하지 않습니다.
- **사유:** 파일 쓰기 완료를 기다리면 프론트엔드의 실시간 타이핑 효과(UX)를 구현할 수 없으며, 치명적인 시스템 에러(Stderr) 로그가 누락될 위험이 있습니다.
- **구현 방식:** 파이썬의 출력 버퍼링을 강제 해제(`PYTHONUNBUFFERED=1`)하여, 에이전트가 생성하는 모든 텍스트를 메모리 버퍼를 거치지 않고 즉각적으로 뱉어내도록 설정합니다.

### 1.2. 조종석(Control Center) 기반의 데이터 흐름
- 사용자는 복잡한 터미널 환경을 전혀 알 필요가 없는 완벽한 웹 기반 폐쇄형 시스템을 지향합니다.
- **파이프라인 구조:**
  1. **명령 하달 (Web UI):** 사용자가 대시보드에서 모델 선택 및 프롬프트를 입력.
  2. **백그라운드 실행 (Spring Boot):** `ProcessBuilder`를 통해 환경 변수를 주입하여 격리된 컨테이너 내에서 에이전트 실행.
  3. **실시간 가로채기 (Spring Boot):** 실행된 프로세스의 표준 출력(Stdout) 파이프를 자바 서버가 직접 연결하여 수신.
  4. **이벤트 스트리밍 (SSE):** 수신된 텍스트를 지연 없이 `SseEmitter`를 통해 프론트엔드로 푸시.
  5. **화면 렌더링 (Vanilla JS):** 브라우저에 가상 터미널 형태로 실시간 출력.

## 2. 환경 변수 및 보안 정책 (Environment Setup)

### 2.1. 중앙 집중형 키 관리 (MVP)
- **정책:** 1주차~2주차 MVP 구현 단계에서는 사용자의 설정 피로도(API Key 입력, 환경 구축 등)를 0으로 만듭니다.
- **구현:** 서버 관리자가 `.env` 파일 및 `docker-compose.yml`에 Gemini API 키 등 필수 환경 변수를 1회 세팅하여 통합 구동합니다.

### 2.2. 향후 확장 정책 (BYOK: Bring Your Own Key)
- 추후 B2B/B2C SaaS 형태로 발전할 경우, 사용자 설정 페이지를 신설하여 사용자의 개별 API 키를 입력받고 `ProcessBuilder` 구동 시점에 동적으로 주입하는 방식으로 전환합니다.

## 3. 다중 에이전트 오케스트레이션 (Multi-Agent Orchestration)

- **현재 (MVP 단계):** 1개의 메인 에이전트(Aider)에 집중하여 '실시간 터미널 뷰'의 역동성을 보여주는 데 주력합니다.
- **미래 (확장 모델):** Claude Code, Codex 등 다중 에이전트 동시 구동 시에는 모든 터미널을 중계하지 않습니다.
- **상태 기반 관제:** 네트워크 트래픽 최적화와 직관적인 UX를 위해 각 에이전트의 상태(Idle ➔ Generating ➔ Testing ➔ Needs Approval ➔ Done)를 모니터링하고 제어하는 '슬롯형 지휘 대시보드'로 전환합니다.
